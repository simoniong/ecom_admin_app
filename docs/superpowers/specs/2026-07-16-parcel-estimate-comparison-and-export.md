# 運費差異頁：per-parcel 預估對比 + 對帳導出 — 設計文件

**日期：** 2026-07-16
**狀態：** 已與使用者逐項確認（透過互動式 HTML preview），待實作
**前置：** 2026-07-14 運費匯入與差異報表（`parcels` 表、`ShippingCostCalculator`、`/parcels` 差異報表）

## 1. 目標

現有 `/parcels` 差異報表只在**訂單層級**顯示預估 vs 實際總額，展開只看得到實際包裹明細。使用者無法：
- 對比預估與實際的**計算依據**（分區、重量、費率）
- 看出**分區被算錯**（貨代把包裹算到比收件地址該有的更貴的分區）
- 拿一份足以跟貨代**對帳**的證據

本次補上三塊：per-parcel 預估對比、分區對照、對帳用 Excel 導出。並把不常用的行內編輯收進「編輯模式」開關。

## 2. 核心概念與定義（重要，money-correctness）

**預估是「每訂單一個數」，per-parcel 預估是新定義的東西。** `ShippingCostCalculator` 拿整張訂單的總（商品）重、解析**一個**分區、算出一個凍結數（`orders.estimated_shipping_cost`）。它沒有包裹概念。

定義（使用者已確認）：

- **每包裹預估** = 拿該包裹的**計費重（`billed_weight_g`）**，套用**同一張運價卡**與**訂單的預估分區**（由收件郵編解析），即時重算。公式：`per_kg_rate × billed_weight_kg + flat_fee`（用計算器既有的 `cost_cny_for(version.rates.where(zone:), weight_kg)`）。
- **即時重算**：用**當前**運價卡。若費率事後改過，會與當初凍結的 `estimated_shipping_cost` 有落差；若訂單郵編缺失或運價卡未涵蓋，預估欄留空。使用者已接受此取捨。
- **訂單總差異拆解**（兩段，使用者已確認命名）：
  - `折包代價` = Σ(每包裹預估) − 訂單預估。來源主要是「拆成 N 包 → 掛號/操作固定費被收 N 次，而訂單預估只算 1 次」＋計費重合計 vs 商品重的差。單包裹時 ≈ ¥0。
  - `物流商可能超收` = 實際 − Σ(每包裹預估)。即「按牌價這些包裹本該多少 vs 實際被收多少」。
  - 恆等式：`實際 − 訂單預估 = 折包代價 + 物流商可能超收`。
- **分區對照**：
  - `預估分區` = 訂單層級，由收件郵編經 `ShippingZonePostalRule` 解析（`ShippingCostCalculator#resolve_zone`）。不分區國家為「—」。
  - `帳單分區` = 每包裹的 `parcels.zone`（帳單「分区」）。
  - 兩者不符 → 標紅、掛「⚠ 分區不符」，並在訂單列掛警示 badge。這是「供應商分錯區」的偵測。

## 3. UI（`/parcels` 訂單差異分頁）

版面已由 HTML preview 定案。每張訂單：

1. **訂單列**：訂單號（分區不符時掛 `⚠ 分區不符` badge）· 預估 · 實際 · 差異 · 差異% · 包裹數。金額**以 CNY 為主、USD 為輔**（¥ 粗體，$ 小字）。
2. **展開後**：
   - **訂單預估依據**一行：🇦🇺 國家 · 分區 X · 商品總重 W kg · 重量帶 · 費率 → = ¥估（$估）。
   - **包裹明細表**，每列：包裹ID · 渠道 · **分區（預估 X / 實際 Y，不符標紅）** · 計費重 · **預估(¥/$)** · **實際(¥/$)** · **差異(¥/$)** · 差異%。金額 CNY 為主。行內編輯（輸入框/存/刪）預設隱藏。
   - **包裹合計列**（tfoot）：計費重合計（對比商品預估重，較重標紅 +Δ）· 預估加總（多包裹時註「含 N× 固定費，較訂單預估 +¥…」）· 實際加總 · 差異加總。
   - **總結欄**：「為什麼差 +¥…？」= `折包代價` + `物流商可能超收`（各帶 $）。分區不符時標題紅字「⚠ 分區被算錯」並註明區別。
3. **編輯模式**：頁面頂端一個「編輯模式」開關，關閉時隱藏所有行內編輯控制項（輸入框/存/刪），開啟才顯示。

## 4. Excel 導出（對帳用）

**用途：跟物流商對帳/理論。** 一個「導出 Excel」按鈕（沿用專案既有 `caxlsx`）。**一列一個包裹**，**主幣別 CNY，不含匯率欄**（帳單即人民幣）。欄位順序：

**A. 定位貨票**：訂單編號(PKS#) · 包裹識別號 · 內部單號 · 貨運單號 · 發貨時間
**B. 目的地與分區**：國家 · 客戶郵編 · 預估分區 · 帳單分區 · 分區是否相符(Y/N)
**C. 重量與渠道**：實際重量(g) · 計費重(g) · 物流渠道
**D. 費用明細(¥)**：運費 · 掛號費 · 稅金 · 偏遠費 · 操作費 · 實際總運費
**E. 對帳結論**：預估運費(¥) · 差異(¥) · 差異%

- 客戶郵編來自訂單收件地址（`order.shopify_data["shipping_address"]["zip"]`）—— 分區的判定依據。
- 預估分區/預估運費即時重算；無法估時留空。
- 導出範圍：跟隨畫面當前的日期/篩選。

## 5. 實作要點

- **`ShippingCostCalculator`**：抽出可重用的原語 —— 給定 order 解析出（version, zone, 商品總重, 訂單預估），以及「給定 zone 與 weight_kg 回傳 CNY」。per-parcel 預估 = 對每個 parcel 用其 `billed_weight_g` 呼叫此原語。回傳結構要含分區與費率，供 UI 的「預估依據」行顯示。
- **`ParcelsController#index`**：對每張訂單計算 per-parcel 預估、分區對照、兩段拆解；避免 N+1（預先載入 parcels、rate card version、postal rules）。
- **導出**：新 action（如 `GET /parcels/export.xlsx`），owner 或有 `parcels` 權限者可用，跟隨篩選。
- 金額一律 BigDecimal；CNY 為帳單原幣，USD 由 `fx_rate_snapshot`（實際）或當前 `cost_fx_rate`（預估）換算。

## 6. 測試（依專案規範：RSpec + FactoryBot，無 mock，真實 DB，95%+）

- per-parcel 預估：計費重套牌價正確；分區解析正確；`折包代價 + 物流商超收 = 實際 − 訂單預估` 恆等式；郵編缺失/未涵蓋時預估留空。
- 分區不符偵測：預估區 ≠ 帳單區時 flag=true。
- 導出：欄位齊全、順序正確、CNY 值取自帳單、預估分區/郵編正確、跟隨篩選、無匯率欄。
- 編輯模式：預設隱藏、開啟顯示（system spec）。
- Mutation 驗證關鍵：折包代價/超收定義、分區不符判定、導出的分區欄取值。

## 7. 已知限制

- 預估依當前運價卡即時重算，非凍結值（見 §2）。歷史費率變更會造成落差。
- 預估分區為訂單層級（單一郵編），同訂單所有包裹共用；帳單分區為每包裹。
